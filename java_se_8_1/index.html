    Absztrakt osztályok és interfészek

    Bevezetés

    Az objektumorientált programozás alapját az osztályok használata képezi, amelyek lehetővé teszik a kódbázis strukturált felépítését és az adatok és viselkedések hatékony kezelését. Azonban az eddig megismert osztályokon túl további osztály típusok is rendelkezésünkre állnak, hogy mélyebben kifejezzük az öröklődési és viselkedési kapcsolatokat a programokban. Ebben a fejezetben mélyrehatóan megvizsgáljuk az absztrakt osztályok és interfészek szerepét és előnyeit a Java nyelvben.

    Absztrakt osztály

    Az absztrakt osztályok speciális osztályok, amelyekben lehetnek olyan metódusok, melyekhez nincs kód rendelve, csak a definíciót írjuk elő. Nyilvánvaló, hogy egy ilyen osztályt nem tudunk példányosítani, hiszen a nem implemetált metódusokat nem tudjuk futtatni. Egy absztrakt osztályból csak leszármaztatni tudunk osztályokat, amikben a nem implementált metódusokat megvalósítjuk az override kullcsszóval. Ezeket a leszármaztatott osztályokat már példányosíthatjuk. Egyszerűen megfogalmazva az absztrakt osztályok valamiféle szerződések, amelyek a származtatott osztályok számára előírnak bizonyos szabványos viselkedést.

    Nézzük az alábbi osztály hierarchiát:

    class Vehicle {
    private String brand;
    private int year;

    public Vehicle(String brand, int year) {
        this.brand = brand;
        this.year = year;
    }

    public void start() {
        System.out.println("A jármű elindult.");
    }

    public void stop() {
        System.out.println("A jármű leállt.");
    }

    public void displayInfo() {
        System.out.println("Márka: " + brand + ", Év: " + year);
    }
}

class Car extends Vehicle {
    private int numberOfDoors;

    public Car(String brand, int year, int numberOfDoors) {
        super(brand, year);
        this.numberOfDoors = numberOfDoors;
    }

    public void honk() {
        System.out.println("A gépkocsi dudál.");
    }
}

class Motorbike extends Vehicle {
    private boolean hasSidecar;

    public Motorbike(String brand, int year, boolean hasSidecar) {
        super(brand, year);
        this.hasSidecar = hasSidecar;
    }

    public void wheelie() {
        System.out.println("A motor megemelkedik a hátsó kerékén.");
    }
}

public class Main {
    public static void main(String[] args) {
        Car car = new Car("Toyota", 2022, 4);
        Motorbike motorbike = new Motorbike("Honda", 2021, false);

        car.displayInfo();
        car.start();
        car.honk();
        car.stop();

        System.out.println();

        motorbike.displayInfo();
        motorbike.start();
        motorbike.wheelie();
        motorbike.stop();
    }
}

    Ebben a példában a Vehicle osztály általános jármű tulajdonságokat tartalmaz, mint a márka és az év. A Car osztály és a Motorbike osztály mindkettő a Vehicle osztályból származik, és specifikus tulajdonságokkal és metódusokkal rendelkeznek. A Car osztály példányai az autóhoz kapcsolódó tulajdonságokkal rendelkeznek, mint például az ajtók száma, míg a Motorbike osztály példányai a motorhoz kapcsolódó tulajdonságokkal, mint például a hátsó kerék emelkedése.

    Ebben a példában a Vehicle osztályból történő objektum létrehozásnak nincs értelme, hiszen egy elvont fogalom. A valódi objektumok mindig a Car vagy Motorbike osztályokból történnek példányosításra. A Vehicle osztályt, azért hoztuk létre, hogy a Car és Motorbike osztály közös tulajdonságait és metódusait kiemeljük egy általános formába. Ha olyan osztályt hozunk létre amiből nem szeretnénk példányosítani, csak más gyerek osztályok közös tulajdonságait szeretnénk kiemelni, akkor osztály helyett absztrakt osztályt is létrehozhatunk. Az absztrakt osztály abban különbözik a sima osztálytól, hogy nem lehet belőle példányt létrehozni.

    Alakítsuk át a fenti példát, hogy a Vehicle osztály absztrakt osztály legyen.

    abstract class Vehicle {
    private String brand;
    private int year;

    public Vehicle(String brand, int year) {
        this.brand = brand;
        this.year = year;
    }

    public void start() {
        System.out.println("A jármű elindult.");
    }

    public void stop() {
        System.out.println("A jármű leállt.");
    }

    public void displayInfo() {
        System.out.println("Márka: " + brand + ", Év: " + year);
    }
}

// ...

    Absztrakt osztályt az abstract kulcsszóval tudunk létrehozni. Ahogy a példában látjuk, nem kellett módosítani az osztályon vagy a belőle származott osztályon azért, mert absztrakt osztályt csináltunk a Vehicle osztályból még a konstruktor is megmaradt, amit a gyerek osztályokban ki kell elégíteni, azonban a fordító nem engedi nekünk, hogy példányt hozzunk létre az absztrakt osztályból.

    Absztrakt metódusok

    Az absztrakt osztályoknak más előnye is van a sima osztályokkal szemben. Lehetőségünk van absztrakt metódusokat létrehozni. Az absztrakt metódusokat az absztrakt osztályok olyan vázlatként használják, amelyekből az öröklődő leszármazott osztályoknak implementálniuk kell ezeket a metódusokat.

    Az absztrakt metódusokat az abstract kulcsszóval jelöljük meg az absztrakt osztályon belül, és nem tartalmaznak implementációt, csak a metódus fejlécét definiálják. Az absztrakt metódusokat a leszármazott osztályoknak kötelezően felül kell írniuk (implementálniuk), különben a leszármazott osztály is absztrakt osztállyá válik.

    Az absztrakt metódusok hasznosak olyan esetekben, amikor az absztrakt osztályban szeretnénk közös viselkedést deklarálni, de az implementációt a leszármazott osztályokra szeretnénk bízni. Ez lehetővé teszi a leszármazott osztályoknak, hogy a saját specifikus viselkedésüket adják hozzá az absztrakt metódusokhoz.

    Nézzük az alábbi példát:

    abstract class Shape {
    abstract double calculateArea(); // (1)
}

class Circle extends Shape {
    private double radius;

    public Circle(double radius) {
        this.radius = radius;
    }

    @Override
    double calculateArea() { // (2)
        return Math.PI * radius * radius;
    }
}

class Rectangle extends Shape {
    private double width;
    private double height;

    public Rectangle(double width, double height) {
        this.width = width;
        this.height = height;
    }

    @Override
    double calculateArea() { // (3)
        return width * height;
    }
}

public class Main {
    public static void main(String[] args) {
        Circle circle = new Circle(5);
        Rectangle rectangle = new Rectangle(4, 6);

        System.out.println("Circle area: " + circle.calculateArea());
        System.out.println("Rectangle area: " + rectangle.calculateArea());
    }
}
    1. Absztrakt metódus deklaráció
    2. Az absztrakt metódus implementációja a Circle osztályban
    3. Az absztrakt metódus implementációja a Rectangle osztályban

    Ahogy a példán látjuk a Circle és Rectangle osztályok mindkettő a Shape osztályból származnak, és implementálják a saját calculateArea metódusukat a specifikus viselkedés kifejezésére. Ahogy a korábbi Vehicle osztálynál is a Shape osztály absztrakt osztály. A terület számítás nincs értelmezve egy elvont fogalom esetén, ezért a calculateArea() metódusnak nem tudunk implementációt adni az ősosztályon belül, azonban minden alakzat rendelkezik területtel, ezért a konkrét osztályokban már értelmezett a metódus. Ahhoz, hogy biztosítsuk, hogy minden Shape ből származó alakzaton tudjunk területet számolni a metódust absztrakttá tesszük, így a leszármazott osztályoknak felül kell írniuk a metódust.

    Interface

    Az interface nem osztály, hanem egy típus, mely nem tartalmaz semmilyen kódot, csak tulajdonságok és metódusok szignatúráját, definícióját. Az osztálynak, mely implementálja az adott interface-t, kötelezően tartalmaznia kell a tulajdonságokat és meg kell valósítani az interface-ben levő metódusokat. Ezért, ha egy osztály implemetál egy interface-t, biztosak lehetünk benne, hogy tartalmazza azokat a metódusokat és tulajdonságokat, melyet az interface -ben előírtunk ezért tekinthetünk az interface-ekre szerződésként ami előírja bizonyos metódusok jelenlétét.

    Egy osztály több interface-t is implementálhat, ezáltal közel hasonlóan viselkedhet, mintha több szülője lenne, azonban mivel az interface-ban előírt metódusok nem tartalmaznak metódus törzset csak a metódus előírását, így nem találkozunk a többszörös öröklődés okozta problémával.

    Ha egy interface-ben mezőket is definiálunk, akkor azok impliciten publikusak, statikusak és finalok lesznek. Erre azért van szükség, mert az interface-ek nem példányosíthatóak, konstruktoruk sem lehet, így a legvalószínűbb eset az, hogy az adattagokat mindenki számára elérhetőként szeretnénk tenni public, ha nem lehet őket objektum példányhoz kötni, akkor csak az osztályhoz köthetjük őket static, és hogy mindenképp legyenek inicializálva konstruktor hiányában is final.

    interface Task {
    void execute();

    String getType();
}

class CodingTask implements Task {
    @Override
    public void execute() {
        System.out.println("Kódolási feladat végrehajtása...");
    }

    @Override
    public String getType() {
        return "Kódolás";
    }
}

class DesignTask implements Task {
    @Override
    public void execute() {
        System.out.println("Tervezési feladat végrehajtása...");
    }

    @Override
    public String getType() {
        return "Tervezés";
    }
}

class TestingTask implements Task {
    @Override
    public void execute() {
        System.out.println("Tesztelési feladat végrehajtása...");
    }

    @Override
    public String getType() {
        return "Tesztelés";
    }
}

class TaskManager {
    void processTask(Task task) {
        System.out.println("Kezdeti lépések a(z) " + task.getType() + " feladat kezeléséhez.");
        task.execute();
        System.out.println("Feladatkezelés befejezve.");
    }
}

public class Main {
    public static void main(String[] args) {
        Task codingTask = new CodingTask();
        Task designTask = new DesignTask();
        Task testingTask = new TestingTask();

        TaskManager taskManager = new TaskManager();

        taskManager.processTask(codingTask);
        System.out.println();

        taskManager.processTask(designTask);
        System.out.println();

        taskManager.processTask(testingTask);
    }
}

    Ebben a példában van egy Task interface, amelyet a CodingTask, DesignTask és TestingTask osztályok implementálnak. Az interface definiálja az execute metódust a feladatok végrehajtásához és a getType metódust a feladat típusának lekéréséhez. A TaskManager osztály egy processTask metódussal rendelkezik, amelyet bármely feladat típusával használhatunk a feladatok kezeléséhez, mert azok mind megvalósítják a Task interface-t.

    Java 8 újítások

    Fontos megjegyezni, hogy a Java 8 előtti időszakban az interface-k törzs nélküli metódusokat tartalmaztak. Azonban a Java 8 megjelenésével megnyílt a lehetőség arra, hogy egyes metódusokat implementáljunk közvetlenül az interface-ekben is. Ezt a változtatást az indokolta, hogy sok esetben szükség volt új metódusok bevezetésére az interface-ekben, azonbanha az új metódusokat egyszerűen hozzáadták volna az interface-ekhez, akkor a módosított interface-ek nem lettek volna kompatibilek a régi kóddal.

    A default kulcsszó alkalmazása lehetővé teszi az interface-ekben deklarált metódusoknak, hogy saját törzsük lehessen. Ezzel megjelöljük, hogy ezek a metódusok nem absztrakt metódusok, vagyis az implementáló osztályban lehetőség van felülírni őket, de ez nem kötelező. Ha az interface metódusához static kulcsszót rendelünk, akkor az implementáló osztályban nem írható felül, ráadásul a metódusra az interface nevén keresztül is hivatkozhatunk.

    interface Vehicle {
    void start();

    void stop();

    default void honk() {
        System.out.println("Tututu!");
    }
}

class Car implements Vehicle {
    @Override
    public void start() {
        System.out.println("Az autó elindult.");
    }

    @Override
    public void stop() {
        System.out.println("Az autó leállt.");
    }
}

class Motorbike implements Vehicle {
    @Override
    public void start() {
        System.out.println("A motorkerékpár elindult.");
    }

    @Override
    public void stop() {
        System.out.println("A motorkerékpár leállt.");
    }
}

public class Main {
    public static void main(String[] args) {
        Car car = new Car();
        Motorbike motorbike = new Motorbike();

        car.start();
        car.honk();
        car.stop();

        System.out.println();

        motorbike.start();
        motorbike.honk();
        motorbike.stop();
    }
}

    Ebben a példában az Vehicle interface deklarál két absztrakt metódust, start és stop, amelyeket az implementáló osztályoknak (Car és Motorbike) implementálniuk kell. Az interface továbbá tartalmaz egy default metódust, honk, amely alapértelmezett dudahangot ad vissza.

    Mind a Car osztály, mind a Motorbike osztály megvalósítja az Vehicle interface-t. Mindkét osztály implementálja a start és stop metódusokat, és örökli a honk metódust az interface-ből.

    Java 9 újítások

    Azzal, hogy Java 8-ban hozott újításnak köszönhetően az interface-ekbe bekerülhettek a törzzsel rendelkező metódusok is, értelmét veszti, hogy ne lehessen privát metódust definiálni egy interface-en belül. A default és statikus metódusokon keresztül immár a privát metódusok elérhetőek lehetnek, Így a Java 9-es verziójától kezdve lehetőségünk van privát metódust is deklarálni az interface-ekben.

    Absztrakt osztály vagy Interface

    Néha bonyolult lehet eldönteni, hogy az adott probléma megoldásához interface-t vagy absztrakt osztályt használjunk. Az alábbi táblázatban láthatjátok az absztrakt osztály és interface közti különbségeket, amelyeket mentén egyszerűbben tudtuk döntést hozni.

    Interface Absztrakt osztály
    nem tartalmazhat adattagot adattagok és konstansok is definiálhatók
    nem példányosítható nem példányosítható
    nem lehet konstuktora lehet konstruktora
    az osztályok megvalósíthatnak több interface-t, ugyanazt az interface-t megvalósíthatja több osztály. egy osztály csak egy absztrakt osztályból származhat, ugyanaz az absztrakt osztály több osztálynak is az őse lehet
    az interface-t megvalósító osztálynak az összes metódust meg kell valósítania. az absztrakt osztályból származó osztály lehet szintén absztrakt, akkor nem kell minden ősosztálybeli metódust implementálnia. Ha a leszármaztatott osztály nem absztrakt, akkor kötelező az összes absztrakt metódust megvalósítania
    Interface csak interface leszármazottja lehet. Absztrakt osztályból származhat absztrakt vagy nem absztrakt osztály is
    Az interface-ek egy osztály perifériás képességeit határozzák meg. Például egy ember és egy jármű is megvalósíthatja az IMozoghat interface-t. Egy absztrakt osztály az osztály alapvető identitását definálja, a leszármaztatott objektumok azonos típusúak.
    Ha a különböző implemetációk csak a metódusok szignatúrájában egyeznek, a megvalósításban lényegesen különböznek, inkább interface-t használjunk Ha a különböző implementációk hasonlóan viselkednek, inkább az absztrakt osztály használatos.
